# Sensitivity Analysis

No production sensitivity campaign has been conducted in this repository. This
section defines the analyses that independent validation should run against
controlled data before relying on the package for production use.

## Objectives

Sensitivity analysis should determine whether capital outputs respond
directionally and proportionately to changes in scenario tails, liquidity
horizon assignment, NMRF stress losses, PLA/backtesting observations, and policy
parameters. It should also identify cliffs, discontinuities, and unsupported
regions.

## Required Analyses

| Analysis | Perturbation | Expected review focus |
| --- | --- | --- |
| ES tail severity | Scale the worst 1 percent, 2.5 percent, and 5 percent scenario losses by controlled factors. | ES monotonicity, tail-weight effect, estimator behavior around fractional tails. |
| Scenario count | Run matched datasets with smaller and larger scenario counts. | Stability of empirical ES, interpolation behavior, and minimum-data handling. |
| Liquidity horizon assignment | Move selected risk factors between 10, 20, 40, 60, and 120 business-day categories. | Nested-vector construction, LHA ES monotonicity, and short-maturity/index helper effects. |
| Risk-class concentration | Concentrate losses in one risk class versus diversified risk classes. | IMCC constrained/unconstrained terms and diversification limits. |
| Reduced set | Vary reduced-set contribution coverage around the 75 percent diagnostic threshold over 60 observations. | Floor behavior and diagnostic sensitivity to contribution quality. |
| Stress-window severity | Shift supplied historical loss/severity vectors and window lengths. | Stress-period selection stability and selected-window explainability. |
| NMRF Type A / Type B mix | Reclassify controlled factors between modellable, Type A, and Type B. | Capital routing, SES aggregation, Type B correlation parameter sensitivity. |
| NMRF stress method | Compare direct, stepwise, full-revaluation, and max-loss artifact inputs for the same factor. | Method-selection evidence and reconciliation controls. |
| PLA metric | Perturb HPL/RTPL level, dispersion, monotonicity, and outliers. | KS and Spearman zones, policy-window enforcement, and amber/red boundary behavior. |
| Backtesting | Inject APL/HPL exceptions at 97.5 percent and 99.0 percent VaR levels. | Exception counts, missing-data treatment, and prorated short-history thresholds. |
| Supervisory multiplier | Move backtesting exception counts across multiplier table boundaries. | Correct mapping and capital sensitivity to traffic-light changes. |

## Datasets

Use only synthetic or approved controlled datasets. The committed
`capital_run_v1` fixture is the first regression sentinel, but it is not enough
for production validation because it is generated by the same codebase and is
not an independent benchmark.

Independent validation should add:

- hand-built small fixtures with closed-form expectations;
- adversarial edge-case fixtures;
- peer-model or spreadsheet references for major formulas;
- approved anonymized bank fixtures only after data-governance controls exist.

## Acceptance Evidence

Each sensitivity run should record:

- fixture identity and input hashes;
- policy profile and policy hash;
- code version and package version;
- parameter perturbation table;
- expected directional response;
- actual response and tolerance rationale;
- explanation for any non-monotonic or discontinuous behavior.

The evidence should link back to `06_change_history.md` when a sensitivity
finding causes a material model change.
