# Repository controls

This document defines the repository-control baseline for `tomanizer/frtb-capital`.
It is audit evidence for the prototype workspace and does not make the software
approved for regulatory reporting.

## Protected branch baseline

`main` is the protected release branch. Configure it with these controls:

| Control | Required setting |
|---|---|
| Pull requests | Require a pull request before merging. |
| Required reviews | Require at least one approving review. |
| Code owners | Require review from `.github/CODEOWNERS` for owned paths. |
| Conversations | Require all conversations to be resolved before merge. |
| Required checks | Require `quality-control` and `ci-required`. `ci-required` aggregates the PR-relevant CI jobs: `docs`, `lint`, `typecheck`, `build`, `test (3.11)`, `dependency-audit`, `sbom`, `examples`, and `notebooks`. Scheduled and manual compatibility runs cover `test (3.12)` and `test (3.13)`. |
| Force pushes | Disabled for `main`. |
| Deletions | Disabled for `main`. Branch deletion after merge is handled by repository auto-delete for merged PR head branches, not by allowing deletion of `main`. |
| Admin bypass | Disabled for normal work. Emergency bypass must be documented in an issue or incident record and followed by a corrective PR. |

Repository settings should also enable automatic deletion of merged pull-request
head branches. That setting deletes feature branches after merge without
weakening protection on `main`.

## Signing and attestation policy

Release integrity is established through:

- Annotated release tags created from the final `main` commit.
- GitHub artifact attestations generated by `.github/workflows/release.yml`
  using OpenID Connect and `actions/attest-build-provenance`.
- A SHA-256 manifest for wheels, source distributions, and the suite SBOM.

If maintainers have a configured GPG or SSH signing key, release tags should be
cryptographically signed. Unsigned annotated tags are allowed only for prototype
releases when the GitHub artifact attestation is present and the release notes
state that tag signing has not yet been enabled for the maintainer account.

Do not publish a GitHub release unless the release workflow produced:

- wheel and source-distribution artifacts under `dist/release/`,
- `dist/sbom/frtb-capital.cdx.json`,
- `dist/release/SHA256SUMS`,
- `dist/release/release-checksums.json`,
- GitHub artifact attestations for the release artifacts.

## CODEOWNERS

`.github/CODEOWNERS` assigns default ownership to `@tomanizer` and explicitly
covers:

- `packages/frtb-ima/src/`,
- `packages/frtb-ima/tests/`,
- `packages/frtb-ima/docs/requirements/`,
- IMA regulatory documentation and source manifests,
- suite ADRs and model-documentation packs.

Add package-specific owners when scaffolded packages move into active
calculation implementation.

## Audit evidence capture

Capture current repository settings with:

```bash
make repo-controls-snapshot
```

The target writes a JSON snapshot to:

```text
dist/repo-controls/frtb-capital-main-repo-controls.json
```

It records the repository metadata, `main` branch-protection settings, and
paginated rulesets visible to the authenticated GitHub CLI user. Override the
defaults when auditing a fork or non-`main` branch:

```bash
make repo-controls-snapshot REPO=owner/name BRANCH=release/v0.1
```

If repository permissions do not allow reading a setting, the snapshot records a
structured unavailable-endpoint entry. Attach that output to the release issue or
governance evidence pack and request an owner/admin export.

Equivalent direct commands are:

```bash
gh api repos/tomanizer/frtb-capital
gh api repos/tomanizer/frtb-capital/branches/main/protection
gh api repos/tomanizer/frtb-capital/rulesets
```
