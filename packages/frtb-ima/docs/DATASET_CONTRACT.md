# Dataset Contract

This repository includes a committed synthetic fixture at
`tests/fixtures/capital_run_v1/`. It is a regression fixture for development,
not market data, client data, regulatory evidence, or a regulatory reporting
submission.

The fixture is generated by:

```bash
make fixtures
```

which runs:

```bash
python scripts/generate_fixture.py --seed 42 --output tests/fixtures/capital_run_v1
```

The generated files are committed so integration tests can run without network
access, external data stores, pandas, DuckDB, SQLite, Parquet, or proprietary
vendor feeds.

## Boundary

The canonical fixture is IMA-native. It mirrors the package inputs that the
capital layer already consumes:

- `RiskFactorDefinition`
- `RFETEvidence`
- `ScenarioCube`
- `HistoricalStressSeries`
- NMRF method evidence and upstream stress artifacts
- PLA and backtesting vectors

It deliberately does not use ISDA SIMM CRIF column names as its primary schema.
SIMM/CRIF is useful for margin and adapter design, but it is not the canonical
shape for FRTB IMA scenario P&L and NMRF valuation artifacts in this package.
Bloomberg, LSEG/Refinitiv, Reuters, ISDA, and client-specific field mappings
belong in future adapter documentation, not in the core regression fixture.

## Files

`tests/fixtures/capital_run_v1/` contains:

- `manifest.json`: schema version, generator version, generator command,
  SHA-256 checksums, and sign conventions.
- `params.json`: run-level parameters used to produce expected outputs,
  including regime, as-of date, ES confidence level, RFET thresholds, PLA and
  backtesting windows, and stress-period window length.
- `risk_factors.csv`: one row per risk factor with `name`, `risk_class`,
  `liquidity_horizon`, `bucket_id`, and `currency`.
- `rfet_observations.csv`: real-price evidence rows with risk-factor name,
  observation date, source, qualitative pass flag, and bucket ID.
- `scenario_metadata.csv`: scenario IDs, dates, and set type for the current
  scenario cube.
- `scenario_cube.npz`: dense current scenario P&L cube with shape
  `(scenario, position, risk_factor)`.
- `stress_history_metadata.csv`: source metadata for supplied historical
  stress histories by risk class.
- `stress_histories.npz`: one historical loss vector, date vector, and scenario
  ID vector per risk class.
- `nmrf_evidence.json`: pre-valuation method evidence for the fixture NMRFs.
- `nmrf_artifacts.npz`: upstream NMRF stress losses and scenario IDs for each
  NMRF artifact.
- `pla_bt_vectors.npz`: APL, HPL, RTPL, VaR, and observation-date vectors.
- `expected_outputs.json`: golden scalar outputs and exact categorical outputs
  used by the integration test.

## Sign Conventions

The fixture manifest declares conventions per array group. The key conventions
are:

- Scenario cube, stress history, and NMRF artifact losses use
  `positive_loss`.
- APL, HPL, and RTPL use `positive_profit`.
- VaR vectors use `positive_magnitude`.
- Dates and scenario IDs are identifiers and have no sign convention.

Tests must respect these conventions. Do not convert signs inside fixture
loading helpers to make a test pass.

## Golden Outputs

`expected_outputs.json` stores float outputs as:

```json
{
  "value": 123.45,
  "rtol": 1e-9,
  "atol": 1e-9
}
```

Tests compare these values with `numpy.testing.assert_allclose`. Categorical
outputs, such as RFET classifications, NMRF methods, PLA zone, reconciliation
status, and selected stress-period IDs, are compared exactly.

The expected outputs are generated by the same deterministic fixture script.
This is intentional: the fixture is a drift gate for the current calculation
contract, not an independent regulatory benchmark.

## Regeneration Rules

When changing fixture schema or calculation semantics:

1. Update `scripts/generate_fixture.py`.
2. Run `make fixtures`.
3. Review diffs in both input files and `expected_outputs.json`.
4. Run `make check`.

If only generated timestamps or checksums changed, the deterministic NPZ writer
is broken and should be fixed before the fixture is accepted.
