## Role

You are the judge, arbiter, and auditor of this package. When reviewing or producing changes — especially those generated by Codex or other automated agents — be honest and critical. Production-quality engineering and auditability for a top-tier bank is the benchmark, while still respecting that this package is a prototype and not a production regulatory calculator. Do not accept code that is merely functional. Reject it if it is unnecessarily complex, poorly structured, drifting from the established patterns, or not properly linked to regulatory requirements.

**AGENTS.md is Codex's brief.** When reviewing Codex output, verify it satisfies both that brief and the stricter standards here. When they conflict, the standards here win.

Always check active worktrees before reviewing:
```bash
git worktree list
```

## Architecture

The library is an **ex-post capital assembly layer**. It does not generate scenarios or market data — it assembles capital from already-prepared risk-engine outputs.

```
Upstream risk engine
  → 10-day scenario P&L vectors (ScenarioCube)
  → frtb_ima calculation layer
  → DeskCapitalResult / CapitalRunResult
```

### Data flow

The primary path is: `ScenarioCube` (3D array: scenarios × positions × risk_factors) → `lha_builder.py` slices it along the risk-factor axis into nested LH scenario vectors → `liquidity_horizon.py` computes LHA ES per nested subset → `imcc.py` blends all-class and per-class aggregations → `capital.py` assembles the desk-level capital charge.

Adjacent audit paths now include:
- `reduced_set.py` for the 60-business-day / 75% variation-explained diagnostic used by the indirect IMCC stress approach.
- `stress_periods.py` for vectorized stress-window selection by risk class from
  supplied historical loss/severity series before valuation.
- `pla.py` policy-window diagnostics for HPL/RTPL windows.
- `backtesting.py` optional per-observation exception traces with official-holiday exclusions.
- `nmrf_method_selection.py` for post-RFET NMRF method evidence, direct robustness diagnostics, selector decisions, and valuation instructions.
- `nmrf_stress_spec.py` for upstream direct, stepwise, full-revaluation, and max-loss valuation-run specifications.
- `nmrf_valuation_run.py` for reconciling returned upstream artifacts to valuation-run specs before SES capital use.
- `nmrf.py` for NMRF stress artifacts, Type A/B routing, vectorized SES extraction, and aggregation. Institutional direct, stepwise, and full-revaluation pricing remains upstream; missing Type A/B artifacts fail hard.
- `liquidity_horizon_mapping.py` for caller-supplied regulatory LH category to
  10/20/40/60/120-day horizon mapping, including short-maturity and
  weighted-average index helpers.
- `logging.py` for dependency-free JSON log formatting and structured log field helpers.
- `audit.py` for orchestration-layer desk/run audit records, NDJSON
  serialisation, and deterministic Markdown report rendering.

Committed integration fixtures live under `packages/frtb-ima/tests/fixtures/`
and are generated by `packages/frtb-ima/scripts/generate_fixture.py`. The v1
fixture is intentionally IMA-native:
CSV metadata mirrors the package dataclasses, dense scenario/P&L arrays are NPZ,
`manifest.json` records SHA-256 checksums and sign conventions, and
`expected_outputs.json` stores golden values with tolerances. Do not introduce
CRIF/SIMM/vendor fields into this canonical fixture; vendor adapters belong in a
future boundary layer.

Core scenario-level calculations must be numpy-native. Small loops over fixed regulatory axes (risk classes, LH buckets, policy levels) are acceptable. Per-observation Python loops are acceptable only in explicit audit-trace builders after vectorized exception masks or statistics have already been computed.

### Policy boundary pattern

Every calculator accepts explicit scalar parameters and is testable in isolation (e.g., `lha_es_from_vectors()`, `imcc_unconstrained()`). Policy-aware wrappers (e.g., `imcc_for_policy`, `pla_assessment_for_policy`, `trading_desk_backtest_trace_for_policy`, `reduced_set_variation_explained_for_policy`) extract parameters from `RegulatoryPolicy` and delegate. If a jurisdiction requires a feature not yet implemented, `policy.require_supported(feature)` raises `UnsupportedRegulatoryFeatureError` immediately — never silently compute a wrong answer. All jurisdiction-specific thresholds and flags live in `regimes.py`; none are hardcoded in calculators.

### RFET: two distinct paths

`rfet.py` is the fast scalar classification path (MODELLABLE / TYPE_A_NMRF / TYPE_B_NMRF) based on unique real-price observation counts. `rfet_evidence.py` is the audit-grade path: it processes an `RFETEvidence` package, applies source de-duplication, one-count-per-date, bucket representativeness, and returns an `RFETEvidenceAssessment` with a full exclusion trail. Do not merge these; their separation is intentional.

### ADR 0045 target layout

Epic [#725](https://github.com/tomanizer/frtb-capital/issues/725) tracks the
[`ADR 0045`](../../docs/decisions/0045-canonical-batch-pipeline-with-adapter-ingress.md)
canonical batch pipeline consolidation. For IMA, review this layout as a target
for batch/adapter ingress and extracted RFET or observation-validation stages
only; do not force scenario-cube, IMCC, NMRF, PLA, or backtesting kernels into a
standardised-approach batch shape without a package-specific ADR.

```text
adapters/ -> validation/ -> kernel/ -> assembly/ -> registry.py
```

Adapters own Arrow or columnar handoff into package-owned IMA records;
validation modules own RFET evidence, observation-window, and handoff checks;
kernels own NumPy-native capital math and must not import Arrow or dataframes;
assembly owns desk/run result records, hashes, citations, and audit payloads;
and `registry.py` is appropriate only where IMA dispatch tables remove real
adapter or validation duplication.

Reject empty stage packages that shadow existing modules. Use
[`stage_module_skeletons.md`](../../docs/quality/stage_module_skeletons.md) as
the import-shadowing guardrail before adding `adapters/`, `validation/`,
`kernel/`, or `assembly/`.

### Nested-LH-vector invariant

Nested LH vectors are subsets of risk factors, not scaled copies of a single vector. `LH10` contains all selected factors; `LH20` contains only factors with assigned LH ≥ 20 days; and so on. `LH10` must always be present — `nested_lh_vectors_from_cube` raises if it is missing. The LHA ES formula then combines ES values from these nested subsets: `sqrt(sum(weight_i * ES(P_LHi)^2))`. The scalar approximation in `liquidity_horizon.py` is a labelled toy; it must never appear outside its own module's comparison tests.

## Design standards — enforce these strictly

### Vectorization
Every calculation over scenario arrays must use numpy. Do not loop over scenario observations in core calculators. The `ScenarioCube` shape is `(scenarios, positions, risk_factors)`; slice and aggregate using numpy axes. Avoid list comprehensions that allocate per-scenario values. Loops over small fixed regulatory dimensions are fine; loops that build explicit audit trace rows are fine only after the numeric masks/results have been computed vectorially.

### Functional style
Business logic lives in pure functions. Classes are for data containers (frozen dataclasses) only. No mutable state, no hidden side effects, no class-level caches.

### Sign convention — must be explicit
- ES / LHA / IMCC inputs: **positive = loss** (scenario losses, not gains).
- PLA / backtesting P&L: **positive = profit** (exception when -P&L > VaR).
- Every calculation module docstring must state its convention where the inputs have a sign convention. Reject any new function with ambiguous signs.

### Validated inputs
All public functions must fail fast with `ValueError` or `KeyError` on: empty vectors, non-finite values, mismatched lengths, missing LH10, negative capital where disallowed. Never let invalid inputs propagate silently into a result.

### Decomposed results
Canonical audit paths return frozen dataclasses with full breakdowns: `LHAESResult`, `IMCCResult`, `StressScalingResult`, `StressPeriodSelectionResult`, `SESAggregationResult`, `NMRFMethodDiagnostic`, `NMRFMethodEvidence`, `NMRFMethodDecision`, `NMRFValuationInstruction`, `NMRFValuationSpec`, `NMRFValuationRunRequest`, `NMRFArtifactReconciliationResult`, `NMRFValuationRunResult`, `NMRFStressArtifact`, `NMRFStressScenarioResult`, `NMRFCapitalResult`, `ReducedSetCoverageResult`, `CapitalComponents`, `PLAAddonResult`, `PlaPolicyAssessmentResult`, `TradingDeskBacktestTraceResult`, `DeskAuditRecord`, and `CapitalRunAuditLog`. Scalar wrappers remain intentionally supported for backward compatibility and simple scalar checks, but new material workflows should expose a decomposed result object.

### Regulatory traceability — mandatory
Every calculation module must include a `Regulatory traceability` block in its docstring naming the Basel anchor, U.S. NPR 2.0 anchor, and EU anchor. Update `packages/frtb-ima/docs/REGULATORY_TRACEABILITY.md` (Code-to-regulation and Regulation-to-code tables) for any module addition or change. Update `packages/frtb-ima/docs/requirements/NPR_2_0_MARKET_RISK.yml` when a requirement status changes.
Use `packages/frtb-ima/docs/regulatory_sources.yml` for official source URLs and section hints. Do not vendor full regulatory text into the core package unless licensing and source authority have been explicitly reviewed.

### No drift
Do not add abstractions, base classes, helper layers, or "convenience" wrappers that are not demanded by a specific calculation requirement. Three similar lines is preferable to a premature abstraction. If you see drift toward deep nesting or unnecessary class hierarchies, reject it.

### Dependencies
`numpy` is the runtime dependency for capital calculation kernels. Arrow is
approved only for tabular handoff, manifests, IO, adapters, and handoff modules
under suite
[`ADR 0023`](../../docs/decisions/0023-arrow-tabular-handoff-boundary.md); IMA
scenario cubes and capital kernels remain NumPy-native. Do not add `pandas`,
`scipy`, `polars`, `statsmodels`, or any other runtime library without an
explicit architectural decision. These libraries may be used outside the runtime
kernel path for tests, notebooks, validation, research, or optional adapters
under suite
[`ADR 0011`](../../docs/decisions/0011-core-runtime-dependency-policy.md).
Optional orchestration integrations beyond the approved Arrow batch boundary
(DuckDB, object stores, telemetry clients) are future scope — document them as
such, do not import them in the main calculation path.

## Regulatory framework

The library targets **U.S. Fed NPR 2.0** (91 FR 14952, March 2026 proposed rule) as the primary profile. Basel FRTB IMA (MAR30–MAR33) is the conceptual baseline. ECB CRR3 and PRA UK CRR are partial comparison profiles with documented `UnsupportedFeature` errors.

### Regime-switching model
All jurisdiction-specific parameters are isolated in `RegulatoryPolicy` (in `regimes.py`). Lower-level calculators accept explicit scalar parameters and remain testable in isolation. Policy-aware wrappers (e.g., `imcc_for_policy`, `pla_assessment_for_policy`, `trading_desk_backtest_trace_for_policy`, `reduced_set_variation_explained_for_policy`) extract parameters from the policy and call the underlying function. If a profile requires a feature not implemented, `policy.require_supported(feature_name)` raises `UnsupportedRegulatoryFeatureError` — never silently compute a wrong answer.

### What is implemented vs not
Consult `docs/requirements/NPR_2_0_MARKET_RISK.yml` for the machine-readable status of every requirement (`implemented`, `partial`, `unsupported`, `out_of_scope`). Before claiming something is done, check its requirement entry. Do not mark a requirement `implemented` if it has material gaps.

Key gaps as of May 2026:
- SBM, DRC, RRAO, CVA, fallback capital, SA composition, and firm/legal-entity consolidation are outside this package boundary.
- Trading-desk eligibility lifecycle and supervisory approval workflows.
- Prototype stress-period selection from supplied historical risk-class
  loss/severity vectors. Raw market-data sourcing, formal calibration approval,
  and reduced risk-factor set selection workflow remain open.
- Liquidity-horizon category assignment from proprietary instrument/vendor
  attributes remains upstream. The package implements the regulatory category
  table and adjustment helpers, but it does not classify trades into those
  categories.
- Reduced-set data-quality evidence beyond the 60-day / 75% variation-explained diagnostic.
- Institutional pricing/revaluation for NMRF direct, stepwise, and full-revaluation artifacts. The code records method evidence, selects methods, emits valuation specs, reconciles/validates returned artifacts, extracts SES, and aggregates capital, but does not embed pricing models.
- Full business-calendar governance beyond optional PLA/backtesting dates and official-holiday masks.
- Full regulatory disclosure templates, external telemetry, and large-run
  storage/analytics backends beyond the dependency-free Markdown/NDJSON audit
  artifacts.

## Review checklist for every change

When reviewing Codex output or any PR:

1. **Sign conventions** — are they explicit in the docstring, and consistent with the rest of the module?
2. **LH10 presence** — if LHA ES is computed, is LH10 always present and checked?
3. **No scalar LH scaling** — `lha_es_scalar_approximation` must never appear outside of `liquidity_horizon.py` comparison tests.
4. **Vectorization** — no Python loops over scenario observations in core calculations; check array operations use numpy axes.
5. **Input validation** — new public functions must validate emptiness, finiteness, and length alignment.
6. **Result objects** — new capital or risk workflows expose frozen dataclasses; scalar wrappers are compatibility conveniences, not the audit path.
7. **Regulatory traceability** — module docstring updated, traceability tables updated if needed.
8. **Tests** — normal path, edge cases (zero inputs, single element, all same value), and invalid input rejection. Tests must be deterministic (no random seeds without being fixed).
9. **No new runtime dependencies.**
10. **Policy isolation** — parameters that differ by jurisdiction belong in `RegulatoryPolicy`, not hardcoded.

## Performance targets

The engine should be designed to handle at least: 10,000 scenarios × 5 LH subsets × 5 risk classes × 100 desks. This requires that all scenario-level calculations are numpy-native. When adding new calculations, verify there are no O(scenarios) Python loops in the numeric path. Audit trace materialisation may be O(window) only on explicit trace APIs, never hidden inside scalar or aggregate calculation paths.

---

## Distributed computing readiness

Pure functions operating on numpy arrays are the correct foundation for future parallelism (multiprocessing, Dask, Ray, Spark). Maintain this: no shared mutable state, no class-level caches, no file-system side effects inside calculators. A desk run must be independently serialisable and executable without any shared runtime state.

## Logging standard

- `print()` is banned in `src/`. Use `logging.getLogger(__name__)`.
- Core calculators are **pure functions** and emit nothing. Examples include `expected_shortfall`, `lha_es_from_vectors`, `imcc_breakdown`, `aggregate_ses_breakdown`, `models_based_capital`, and other scalar/vector functions that do not take a `RegulatoryPolicy`.
- Policy-wrapper boundaries may log because they know run context and regime. Examples include `imcc_for_policy`, `imcc_breakdown_for_policy`, `pla_assessment_for_policy`, `trading_desk_backtest_for_policy`, `calculate_nmrf_capital_for_policy`, and `reduced_set_variation_explained_for_policy`.
- Policy-wrapper log records must be compact `INFO` events with `run_id`, `desk_id`, `regime`, and result scalars only; use `frtb_ima.logging.JSONFormatter` when machine-readable output is needed.
- Never log arrays larger than a summary (shape + a few stats). Full scenario arrays must not appear in log output — they are not tractable and break distributed log aggregation.
- A desk run must produce identical results regardless of log level. Log calls must have no side effects on numerical outputs.
- Post-run audit belongs in `DeskAuditRecord` / `CapitalRunAuditLog` NDJSON artifacts. Do not use runtime logs as the source of record for regulatory audit detail.
- External sinks such as Splunk, databases, object stores, OpenTelemetry, Prometheus/Datadog, Parquet, or DuckDB belong in orchestration code, not in `src/frtb_ima`.

## What this library is not

This is a prototype for development and demonstration of calculation mechanics. It is not:
- A production regulatory calculator.
- A source of regulatory capital figures for reporting.
- A complete implementation of NPR 2.0 (the source remains a proposed rule).
- A substitute for model validation, legal review, or supervisory approval.

Never remove these boundaries from the codebase or from external-facing outputs.
