# Dataset Contract

The RRAO package includes committed synthetic fixture packs under
`packages/frtb-rrao/tests/fixtures/`. They are regression fixtures for
development, not market data, client data, regulatory evidence, or a regulatory
reporting submission.

The committed fixture packs are:

- `rrao_v1/`: static U.S. NPR 2.0 validation fixture with included lines,
  excluded zero-capital lines, expected outputs, and invalid cases.
- `rrao_sample_book_v1/`: generated U.S. NPR 2.0 sample book covering 25
  positions across desks, legal entities, supported evidence types,
  investment-fund inclusion, supervisor-directed inclusion, and exclusions.
- `rrao_eu/`: static EU CRR3 comparison fixture for Article 325u and Delegated
  Regulation (EU) 2022/2328 mappings.
- `rrao_pra/`: static PRA UK CRR comparison fixture for UK CRR Article 325u and
  UK retained Delegated Regulation (EU) 2022/2328 mappings.

## Boundary

The canonical RRAO fixture shape is RRAO-native. It mirrors the package inputs
that the capital layer already consumes:

- `RraoCalculationContext`
- `RraoPosition`
- `RraoSourceLineage`
- `RraoBackToBackMatch`
- `RraoInvestmentFundDescriptor`

CRIF, FNet, and vendor-shaped rows are adapter inputs, not the canonical
regression fixture schema. Vendor field mappings belong in adapter
documentation and tests. Core fixtures should continue to use RRAO-native
records and enum values so classification, exclusions, allocation, audit,
batch, and Arrow paths all exercise the same domain contract.

## Files

`packages/frtb-rrao/tests/fixtures/rrao_v1/` contains:

- `positions.json`: supported U.S. NPR 2.0 context and canonical positions.
- `expected_outputs.json`: expected classifications, line add-ons, excluded
  lines, subtotals, total capital, warnings, citations, and deterministic
  hashes.
- `invalid_cases.json`: unsupported or invalid evidence cases that must fail
  before a successful result is returned.
- `loader.py`: test-only loader that converts fixture JSON into frozen RRAO
  dataclasses.
- `README.md`: case-level intent and citation ids.

`packages/frtb-rrao/tests/fixtures/rrao_sample_book_v1/` contains:

- `positions.json`: generated U.S. NPR 2.0 sample-book context, fixture notes,
  and canonical positions.
- `expected_outputs.json`: golden result payload for the generated sample book.
- `manifest.json`: schema version, generator command, generator version,
  profile, calculation date, position count, total RRAO, and SHA-256 checksums.
- `README.md`: position groups, expected totals, regeneration command, and
  profile compatibility notes.

`packages/frtb-rrao/tests/fixtures/rrao_eu/` contains:

- `positions.json`: supported EU CRR3 comparison context, positions, and
  expected fixture-level totals for profile tests.
- `README.md`: Article 325u / Delegated Regulation (EU) 2022/2328 coverage.

`packages/frtb-rrao/tests/fixtures/rrao_pra/` contains:

- `positions.json`: supported PRA UK CRR comparison context, positions, and
  expected fixture-level totals for profile tests.
- `README.md`: UK CRR Article 325u / UK retained Delegated Regulation
  (EU) 2022/2328 coverage.

## Fixture Generation

`rrao_sample_book_v1/` is generated by:

```bash
uv run python packages/frtb-rrao/scripts/generate_fixture.py \
  --output packages/frtb-rrao/tests/fixtures/rrao_sample_book_v1
```

From `packages/frtb-rrao`, the equivalent command is:

```bash
uv run python scripts/generate_fixture.py --output tests/fixtures/rrao_sample_book_v1
```

The generator serializes its deterministic `POSITIONS` and `CONTEXT`,
calculates capital with `calculate_rrao_capital`, writes expected outputs, and
records SHA-256 checksums in `manifest.json`.

The `rrao_v1`, `rrao_eu`, and `rrao_pra` fixture packs are static validation or
comparison fixtures. Update those files directly when their fixture contract
changes.

## Production Input Lineage

Production-style RRAO runs should carry stable source lineage on every accepted
position. The canonical dataclasses expose `RraoSourceLineage` with:

- source system,
- source file,
- source row id,
- optional source-column map.

The Arrow handoff path carries the same lineage through `lineage_source_system`,
`lineage_source_file`, and optional `lineage_source_row_id` columns. Batch
builders preserve `source_hash`, `handoff_hash`, adapter diagnostics,
lineage-present flags, and per-row citations so audit output and replay can
distinguish source content, normalized table content, and package-owned
calculation inputs.

Zero-capital exclusions remain auditable output lines. Do not drop excluded
input rows during adapter normalization or fixture loading.

## Arrow Table Handoff

High-volume tabular inputs should enter through:

| Input family | Normalize | Build batch |
| --- | --- | --- |
| RRAO positions | `normalize_rrao_arrow_table(...)` | `build_rrao_batch_from_arrow(...)` |

The Python `RRAO_ARROW_COLUMN_SPECS` tuple is the source of truth for the
client-facing input table. The machine-readable schema artifact is
`docs/schemas/input_table/frtb_rrao.positions.schema.json`.

The Arrow spec accepts documented aliases, normalizes to canonical snake_case
names, and enforces required columns, null policy, and logical type before
NumPy batch construction. Calculation kernels consume package-owned batches,
not `pyarrow.Table` objects.

### Required Column Families

RRAO position tables carry:

- position id: `position_id`,
- source row id: `source_row_id`,
- desk id: `desk_id`,
- legal entity: `legal_entity`,
- gross effective notional: `gross_effective_notional`,
- currency: `currency`,
- evidence type: `evidence_type`,
- evidence label: `evidence_label`,
- lineage source system and file: `lineage_source_system`,
  `lineage_source_file`.

Optional classification and exclusion evidence includes:

- classification hint: `classification_hint`,
- exclusion reason: `exclusion_reason`,
- exclusion evidence id: `exclusion_evidence_id`,
- back-to-back match group id: `back_to_back_match_group_id`,
- back-to-back matched position id: `back_to_back_matched_position_id`,
- supervisor directive id: `supervisor_directive_id`,
- underlying count: `underlying_count`,
- path-dependence, maturity, strike/barrier, multiple-strike/barrier, and CTP
  hedge flags: `is_path_dependent`, `has_maturity`,
  `has_strike_or_barrier`, `has_multiple_strikes_or_barriers`,
  `is_ctp_hedge`,
- comma-separated citation identifiers: `citations`.

Optional investment-fund evidence includes:

- investment fund id: `investment_fund_id`,
- section 205 method: `investment_fund_section_205_method`,
- included exposure type: `investment_fund_included_exposure_type`,
- mandate evidence id: `investment_fund_mandate_evidence_id`,
- section 205 evidence id: `investment_fund_section_205_evidence_id`,
- investment-fund gross effective notional:
  `investment_fund_gross_effective_notional`,
- included exposure ratio: `investment_fund_included_exposure_ratio`,
- look-through-available flag: `investment_fund_look_through_available`,
- mandate-allows-RRAO-exposures flag:
  `investment_fund_mandate_allows_rrao_exposures`.

Nested descriptor objects are not accepted in the Arrow table. Inputs that use
`unsupported_nested_payload`, `investment_fund_descriptor`,
`back_to_back_match`, or `nested_payload` must be flattened into the explicit
RRAO Arrow columns before `build_rrao_batch_from_arrow(...)`.

## Adapter Inputs

The package exposes optional standard-library adapters for CRIF, FNet, and
generic vendor-shaped rows:

- `adapt_crif_records(...)`
- `adapt_fnet_records(...)`
- `adapt_rrao_records(...)`

Adapters must emit canonical RRAO records, rejected rows, and warnings without
changing the calculation contract. Adapter rejection evidence should remain
auditable and deterministic. Client integrations should prefer the Arrow table
path for high-volume production input.

## Sign And Notional Conventions

RRAO line capital is additive by position. The key input convention is:

- `gross_effective_notional` is the reported gross effective notional used for
  the residual-risk add-on line.
- `currency` identifies the notional currency. Current fixtures use `USD`.
- `notional_source` records where the notional came from, such as reported
  notional or investment-fund descriptor evidence.
- Excluded positions keep their reported notional but produce zero-capital
  audit lines.

Do not flip signs inside fixture loaders or batch builders to make a test pass.
If a vendor feed uses a different notional or sign convention, normalize it in
adapter code and keep the RRAO-native fixture/batch contract explicit.

## Golden Outputs

`expected_outputs.json` files store selected deterministic outputs rather than
full result snapshots. Depending on the fixture, they cover:

- profile id and profile hash,
- input hash,
- total RRAO,
- included line classifications, risk weights, add-ons, reason codes, and
  citations,
- excluded zero-capital lines and exclusion evidence,
- subtotals,
- warnings,
- fixture-level citations.

Tests compare floating-point values with deterministic tolerances and compare
categorical outputs, citations, warnings, and invalid-case errors exactly.
These fixtures are drift gates for the current calculation contract, not
independent regulatory benchmarks.

## Regeneration Rules

When changing generated sample-book schema or calculation semantics:

1. Update `packages/frtb-rrao/scripts/generate_fixture.py` or the generated
   sample-book positions it defines.
2. Regenerate `rrao_sample_book_v1/`:

```bash
uv run python packages/frtb-rrao/scripts/generate_fixture.py \
  --output packages/frtb-rrao/tests/fixtures/rrao_sample_book_v1
```

3. Review diffs in `positions.json`, `expected_outputs.json`, and
   `manifest.json`.
4. Run the affected RRAO tests, for example:

```bash
uv run pytest packages/frtb-rrao/tests/test_rrao_fixture_workflow.py \
  packages/frtb-rrao/tests/test_rrao_arrow_batch.py \
  packages/frtb-rrao/tests/test_rrao_eu_profile.py \
  packages/frtb-rrao/tests/test_rrao_pra_profile.py
```

When changing static fixture packs, update the relevant `README.md`,
`positions.json`, `expected_outputs.json`, `invalid_cases.json`, and loader
files as needed, then run the fixture-specific tests plus
`packages/frtb-rrao/tests/test_rrao_arrow_batch.py` if the Arrow path is
affected.

For broader RRAO changes, run:

```bash
uv run pytest packages/frtb-rrao/tests
```

Before publishing package documentation changes, run `make quality-control`
from the repository root when practical.

If only generated timestamps, checksums without content changes, or incidental
ordering changed, fix fixture determinism before accepting the update.
